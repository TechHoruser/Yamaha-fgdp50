## What this repo is

Desktop looper app for the **Yamaha FGDP-50** finger-drum pad.
**Wails** packages a **Go** backend (looper state machine, StreamDeck
HTTP server, auto-updater) with a **React + TypeScript + Vite**
frontend that owns all real-time audio via the **Web Audio API**.

### Backend ↔ frontend wiring

- Every method on `App` in `apps/desktop/app.go` is exposed to JS at
  `window.go.main.App.<Name>`. We mirror it manually in
  `frontend/src/wailsjs/go/main/App.ts` — **edit both files** when
  adding a binding.
- Frontend polls `GetTracks()` / `IsMetronomeActive()` /
  `GetActiveTrack()` every 300 ms in `useLooperState`. There is no
  push channel — backend mutations show up on the next poll.
- Audio I/O is **entirely browser-side** in `LoopEngine.ts`. The Go
  backend only tracks discrete state (`HasLoop`, `Muted`, etc.). When
  you add a state-changing op (e.g. Merge*, RemoveTrack), the
  frontend must also do the matching audio op (`mergeInto`,
  `destroyTrack`) — they are independent stores that must stay in
  sync.

### Track IDs

- Track IDs are **monotonically increasing** and never reused. After
  removing a track, the next AddTrack returns max+1 — not the freed
  slot.
- `SelectTrack(id)` matches by ID, not slice index.
- `GetActiveTrack()` returns the current track's ID, not its index.
- Tests in `internal/looper/looper_test.go` expect
  `DefaultInitialTracks = 4` and the first IDs to be 1..4 — don't
  break that contract.

### MIDI

- Web MIDI API only (no Go-side MIDI). Available in Wails because
  the underlying WebView is Chromium (WebView2 on Windows). On macOS
  WKWebView, support is partial — `useMidiInput` reports
  `supported=false` and the UI degrades to "MIDI N/D".
- The FGDP-50 sends notes on channel 10. Note numbers 36/38/40
  (kick/snare/hi-hat) are mapped to Record/Play/Stop in `App.tsx`.
  All other notes are surfaced to the caller via the `onNote`
  callback but currently ignored.

### Audio engine notes

- Uses the legacy `ScriptProcessorNode` — works fine, just
  deprecated. AudioWorklet migration is queued but not done.
- Do **not** force `sampleRate` on the AudioContext: macOS only
  allows the hardware rate and forcing 44100 inserts an internal
  resampler that costs latency and quality. `useAudioEngine` passes
  no sampleRate; the context adopts the device rate.
- `latencyHint: 'interactive'` is required — anything else makes the
  looper feel rubbery.
- Output device selection uses Chromium's `AudioContext.setSinkId`
  (not in TS lib types yet — declared inline in `LoopEngine.ts`).

### ASIO / pro-audio (not done)

The user asked for ASIO. Web Audio cannot do ASIO. Implementing it
requires replacing `LoopEngine` with a native Go audio engine using
CGo bindings (PortAudio, miniaudio, or RtAudio). That is a
multi-day refactor with Windows-specific code paths and is out of
scope for any single session. Build deps would also need updating
(Windows already has MinGW via the existing NSIS build, so CGo
should work).

If the user pushes for it next time:
1. Plan a `internal/nativeaudio/` package with an interface that
   matches LoopEngine's surface.
2. Add a runtime toggle in settings (Web Audio | Native).
3. Build a CGo wrapper around miniaudio (header-only, ASIO+WASAPI
   on Windows, CoreAudio on macOS).

## Commands

```bash
# Backend (Go)
cd apps/desktop
go test ./...                                # everything
go test ./internal/looper/...                # just looper
go build ./...                               # compile check

# Frontend
cd apps/desktop/frontend
pnpm typecheck                               # tsc --noEmit
pnpm lint                                    # eslint
pnpm test                                    # vitest run
pnpm build                                   # tsc + vite build → dist/

# Dev (Wails-aware)
wails dev                                    # at apps/desktop
```

The frontend build emits to `apps/desktop/frontend/dist/`; that path
is `.gitignored` except for a `.gitkeep` placeholder that Wails
embed needs. Don't commit built assets.

## Conventions (enforced by hooks)

### Commit messages (commitlint via husky)

- `<type>(<scope>): <subject>`
- **Subject must be lowercase** — the validator rejects start-case
  ("MIDI input" fails, "midi input" passes).
- Valid scopes: `desktop`, `audio`, `streamdeck`, `looper`, `ui`,
  `deps`, `ci`, `docs`. Anything else is rejected.
- Pre-commit + commit-msg hooks run; do **not** bypass with
  `--no-verify` unless the user explicitly tells you to.

### Branch

The agent's working branch is `claude/refactor-api-track-selection-VJSCD`.
All pushes go here. Releases happen on `v*` tags from main; do not
push tags yourself — the local git proxy in the web sandbox returns
HTTP 403 on tag pushes. Ask the user to tag from their machine.

### Adding a Wails-bound method (checklist)

1. Add method to `App` struct in `apps/desktop/app.go`.
2. Add matching `export async function` in
   `apps/desktop/frontend/src/wailsjs/go/main/App.ts` (the file is
   hand-written, not auto-regenerated).
3. If it also lives on `looper.Engine`, add the public method there
   and unit-test it in `internal/looper/looper_test.go`.
4. If it should be reachable from StreamDeck, add to
   `internal/streamdeck/commands.go` + dispatch in `server.go` +
   the same constant in `frontend/.../stream-deck-controller/commands.ts`.

## Release workflow

`.github/workflows/release.yml` runs on `v*` tags:

1. extract version from tag
2. `orhun/git-cliff-action@v4` generates release notes
3. matrix builds Windows installer (NSIS via choco, MinGW CGo) and
   macOS universal `.app` → zip
4. `softprops/action-gh-release@v2` publishes

Past breakage points to keep in mind:
- `taiki-e/install-action` curl install was failing under the local
  proxy.
- `orhun/git-cliff-action@v3` failed because its Dockerfile pinned
  Debian Buster which is EOL (404 on apt repos). `@v4` fixed it.

## Environment quirks (Claude Code on the web)

- Working dir is reset on every fresh session — anything not in git
  is gone.
- `origin` points to a local proxy `http://local_proxy@127.0.0.1:*`.
  Branch pushes work; **tag pushes return HTTP 403** — instruct the
  user to tag from their machine.
- `gh` CLI is unavailable. Use the `mcp__github__*` tools for any
  GitHub interaction (PRs, comments, etc.).
